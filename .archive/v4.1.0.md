# Danbooru Mobile Note Assist v4.1.0 — 작업이 사라지지 않는 노트 도구

오랫동안 마음 한구석에 걸리던 문제 하나를 v4.1.0에서 해결했다. 모바일로 번역 작업을 하다가 탭이 강제 종료되거나, 폰을 잠시 다른 앱으로 옮겼다가 OS가 백그라운드 메모리를 정리하면서 브라우저를 죽이는 경우 — 그 동안 만들어둔 노트가 통째로 사라졌다. 이번 업데이트의 핵심은 그런 상황에서도 작업이 보존되도록 만든 거다.

## 사라지던 작업

스크립트의 기존 동작은 단순했다. ✓ Confirm을 눌러서 Danbooru 서버로 batch를 보내기 전까지 모든 노트는 메모리에만 있었다. PC에서 새로고침이나 탭 닫기 같은 명시적 종료에 대해서는 브라우저가 "Leave site?" 확인 창을 띄워줬지만, 모바일에서는 그게 잘 동작하지 않는다. iOS Safari나 Chrome은 폰 홈 버튼, 앱 스위처, 다른 앱으로 전환, 잠시 후 OS의 메모리 정리 — 이런 자연스러운 흐름에서 확인 창 없이 그냥 페이지를 죽인다. 사용자 입장에서는 노트가 그냥 사라지는 거다.

가장 답답한 케이스는 긴 번역문을 막 입력하다가 알림 한 번 보러 다른 앱 다녀온 사이 노트가 전부 없어진 상황이다. 한 번 겪으면 다음부터는 노트 하나 만들 때마다 ✓ 누르는 방어적 습관이 생기는데, 그러면 multi-note batched Confirm의 의미가 무너진다. 여러 노트를 일관성 있게 다뤄야 좋은 번역이 나오는데, 하나하나 즉시 commit하면 그 흐름이 끊긴다.

## 새 동작

이제 두 가지 시점에 자동으로 작업이 백업된다.

첫 번째는 페이지가 사라지려는 모든 신호 — 탭 닫기, 새로고침, 폰 홈으로 이동, 다른 탭 전환. 두 번째는 사용자가 명시적으로 작업을 끝내지 않은 상태에서 OS가 페이지를 백그라운드로 보내는 순간. 이 시점들에서 현재 만들어둔 노트 / 편집 / 활성 선택 / 작업 모드가 모두 localStorage에 스냅샷된다.

같은 post 페이지를 다시 열면 스크립트가 부팅하면서 그 스냅샷을 감지하고 화면 하단에 안내 토스트를 띄운다 — "Saved draft found (3 notes). Restore your work?" 옆에 [Restore]와 [Discard] 두 버튼이 있고, 사용자가 직접 선택할 때까지 토스트가 자동으로 사라지지 않는다.

[Restore]를 누르면 즉시 활성 모드로 들어가면서 박스가 모두 복원된다. 박스 위치, 입력한 텍스트, soft-delete 상태, 어떤 박스가 활성이었는지까지 그대로. [Discard]를 누르면 스냅샷이 사라지고 깨끗한 상태로 시작한다.

복원된 상태에는 그 사이 다른 사용자가 새로 추가한 server 노트도 자동으로 합쳐진다. 정확히는 — 같은 노트에 대해서는 사용자의 draft 버전이 이기고, draft에 없던 새 server 노트만 합쳐진다. 그러니까 작업하던 흐름을 그대로 이어가면서 동시에 그 사이의 외부 변경도 반영된다.

복원 안내를 띄울지 말지는 명시적으로 사용자 선택에 맡겼다. 자동 복원도 고민했지만, 다른 디바이스에서 작업한 줄 알았던 노트가 갑자기 나타나면 오히려 더 혼란스러울 수 있다. 명시적 선택지가 단순하고 예측 가능하다.

## 안전 장치들

작업이 사라지지 않는 동시에 의도치 않은 데이터 누적은 피해야 한다. 몇 가지 처리:

- **24시간 TTL.** 저장된 draft가 24시간 이상 오래되면 자동으로 폐기된다. 하루 뒤에 같은 post를 열면 복원 안내가 안 뜨고 깨끗하게 시작한다. 너무 짧으면 "어제 작업하던 거 살릴 수 있나" 직관에 어긋나고, 너무 길면 server 상태가 그 사이 변했을 가능성이 커진다. 24시간이 자연스러운 중간.
- **post id별로 분리.** 키는 `dmna_draft_{postId}` 형태라 다른 post의 draft가 섞이지 않는다. 여러 post를 동시에 작업하다가 force-quit 당해도 각 post의 작업이 독립적으로 복원된다.
- **명시적 종료에서는 즉시 정리.** ✓ Confirm 성공 직후, Edit 모드를 직접 끄는 순간, Confirm 진행 중 (이 경우는 서버가 최종 정답이라 draft가 오히려 혼동 유발) — 이 세 경우는 draft가 즉시 사라진다.
- **localStorage 한계 / 사파리 프라이빗 모드.** 저장 공간이 꽉 차거나 시크릿 탭처럼 storage 자체가 막힌 환경에서는 조용히 저장을 포기한다. 토스트로 알람을 띄우지 않는다 — 대부분의 사용자에게는 의미 없는 에러라서 노이즈만 늘릴 뿐이다.

데이터는 사용자 브라우저에만 남는다. Danbooru 서버로 전송되지 않고, 다른 디바이스로 동기화되지 않는다.

## 내부 정리

이번 사이클에는 force-quit 가드와 함께 v4.0 직후 미뤘던 작은 type / 구조 정리도 묶었다. 토스트 레벨 타입을 한 곳으로 모으고, 쓰지 않던 중복 hook 호출을 제거하고, 일부 함수의 반환 타입을 좁히는 정도. 동작 변화는 0이지만 다음 cycle에서 새 기능을 얹을 때 조금 더 깔끔한 기반이 된다.

## 다음 cycle

v4.2에서는 popover 자체의 UX 보강을 작업할 예정이다. 노트 입력 박스 옆에 스타일 추가 보조 popover를 띄우는 기능, 그리고 server에 이미 있는 노트의 편집 이력 페이지로 빠르게 이동하는 [History] 버튼이 들어간다. UI 중심 cycle이라 v4.1.0과 release scope를 분리했다 — v4.1.0은 안정성 보강에 집중하고, UI 변화는 따로 묶는 게 release notes가 명료해진다.

## 인스톨 / 업데이트

기존 사용자는 별도 작업이 필요 없다. Tampermonkey가 다음 자동 업데이트 체크 때 v4.1.0을 가져온다. 새로 설치하려면 README의 인스톨 링크를 따라가면 된다. 사용 방법은 v4.0과 동일하고, 복원 토스트는 실제로 draft가 저장된 상황에서만 등장한다.

조금 더 안심하고 모바일에서 노트 작업을 이어갈 수 있길 바란다.
